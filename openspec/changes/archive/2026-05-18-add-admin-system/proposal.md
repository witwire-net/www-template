## Why

顧客向けの認証基盤（パスキー登録・ログイン・セッション管理）は整備されているが、オペレーターが顧客アカウントを検索・閲覧・操作するための管理画面が存在しない。サポート業務や不正利用調査においてオペレーターが DB を直接操作するのは危険であり、操作の監査証跡も残らない。

Admin Console は顧客向けシステムから完全に分離された独立システムとして構築する。認証はプロダクションと同様の WebAuthn passkey 方式とし、httpOnly cookie でセッションを管理する。admin / operator / viewer のロールベース権限制御と全操作の監査ログを備え、安全かつ監査可能な方法で顧客アカウントを管理可能にする。

## What Changes

- **新規**: Admin Console アプリケーション（SvelteKit full-stack / Node.js adapter-node / Docker コンテナ）
- **新規**: Admin-owned schema（`www-template`）— Prisma ORM Mapper で扱うオペレーターテーブル、passkey credential テーブル、監査ログテーブル
- **新規**: database（`www-template`）への管理用拡張 — accounts に status / session_revoked_after カラム追加、admin_view ビュー、admin_op 管理関数、Admin 用 DB role/grant
- **新規**: Admin 独自の認証基盤 — WebAuthn passkey 認証（プロダクションと同方式）、JWT httpOnly cookie セッション、challenge 管理は Product auth と同じ Valkey infrastructure を共有し、`ADMIN_VALKEY_URL` で明示した別 logical DB 番号で実施
- **新規**: オペレーター passkey 管理（追加・一覧・削除）とオペレーター setup token フロー。初回起動時はオペレーター 0 件、明示 enable flag、有効期限付き bootstrap secret をすべて満たす場合のみ特別な初期セットアップ画面で最初の admin オペレーターを作成し、admin が追加したオペレーターは one-time setup token で初回 passkey 登録する
- **新規**: 監査ログの OpenSearch 連携 — 監査イベントを Production ドメイン index と分離した Admin 専用 OpenSearch namespace にインデックスし、高速検索・集計を可能にする
- **新規**: MVCS + RBAC アーキテクチャ — Model / View / Controller / Service / Infrastructure の層分離
- **新規**: ESLint による Admin 専用 lint ルール（MVCS 層間依存強制、顧客向けパッケージ import 禁止、セキュリティ制約）

## Spec Units

### New Spec Units

- `admin-auth-fe`: Admin Console の認証フロントエンド層 — オペレーターパスキーログイン UI、passkey 管理画面、session ガード、no-store 配信、bootstrap gate 付き初期セットアップ画面
- `admin-auth-be`: Admin Console の認証バックエンド層 — WebAuthn challenge/assertion 検証、JWT httpOnly cookie 発行、オペレーター passkey credential 保存、bootstrap gate 付き初回起動セットアップ、追加オペレーター setup token 管理、rate limit と temporary lock
- `admin-console-fe`: Admin Console の管理フロントエンド層 — Dashboard、アカウント検索・詳細・停止・復旧、監査ログ、設定、オペレーター管理、画面レイアウト、MVCS import 制約
- `admin-console-be`: Admin Console の管理バックエンド層 — Admin-owned schema schema、database 拡張（status カラム・ビュー・関数）、Service 層監査保証、DB 接続管理、migration runner、RBAC 権限モデル、セキュリティ lint 制約

### Modified Spec Units

- `auth-be`: database の `accounts.status='suspended'` と `accounts.session_revoked_after` を顧客向け認証境界に反映する。既存の access token / refresh token pair と refresh rotation 契約を保持したまま、Admin suspend が成功した account は新規 token pair 発行、refresh rotation、既存 bearer access token 認可を拒否される。
- `auth-fe`: suspended account のログイン失敗・既存 token pair 失効を、account existence を漏らさない範囲で明確な案内に接続する。

## Naming

- `ADMIN-AUTH-FE-S###` — Admin Console 認証フロントエンドのシナリオ
- `ADMIN-AUTH-BE-S###` — Admin Console 認証バックエンドのシナリオ
- `ADMIN-CONSOLE-FE-S###` — Admin Console 管理フロントエンドのシナリオ
- `ADMIN-CONSOLE-BE-S###` — Admin Console 管理バックエンドのシナリオ
- `AUTH-FE-S###` / `AUTH-BE-S###` — 既存顧客向け認証 Spec Unit の modified scenario
