## Why

現在の認証システムは、 opaque な bearer トークンを単一セッションとして管理しており、以下の運用上・セキュリティ上の課題がある。

- **セキュリティ**: 長寿命の bearer トークンがリークした場合、有効期限まで悪用され続けるリスクがある。短命なアクセストークンと長寿命なリフレッシュトークンの分離が必要。
- **マルチアカウント**: 複数アカウントを利用するユーザーにとって、都度ログインし直す体験は非効率であり、既存の単一セッション管理では複数アカウント同時ログインが不可能。
- **運用**: トークンの有効期限やローテーションがないため、強制ログアウトやセッション無効化の手段が限定的。

これらを解決するため、JWT アクセストークン（短命）＋リフレッシュトークン（長寿命・ローテーション）の仕組みを導入し、クライアントが複数アカウントのセッションを同時に保持・切り替えできるようにする。

## What Changes

1. **JWT アクセストークンの導入**
   - 従来の opaque bearer トークンを、15分程度の短命 JWT アクセストークンに置き換える。
   - アクセストークンは最小限のクレーム（accountID, sessionID, issuedAt, expiresAt）を含む。
   - バックエンドは保護されたリクエスト毎に JWT の署名と有効期限を検証する。

2. **リフレッシュトークンとローテーション**
   - Valkey に保存された長寿命リフレッシュトークンを導入する。
   - リフレッシュトークンは使用毎にローテーション（単一利用）し、GETDEL で原子消費する。
   - リフレッシュトークンはアカウント＋デバイス/セッション指紋にバインドする。
   - `POST /api/v1/auth/refresh` で有効なリフレッシュトークンを新しいアクセストークン＋リフレッシュトークンと交換する。

3. **リフレッシュトークン TTL の設定**
   - `Config.toml` に `auth.refresh_token_ttl`（`time.Duration`、省略可）を追加する。
   - 未設定またはゼロの場合、リフレッシュトークンは無期限有効。
   - 設定時は最低24時間以上を強制（短すぎる設定を fail-close で防止）。

4. **フロントエンドのトークン有効性チェック**
   - クライアントは JWT ペイロードをデコード（署名検証不要）して `exp` を確認する。
   - アクセストークンが期限切れまたは1分未満で期限切れになる場合、プロアクティブに `/api/v1/auth/refresh` を呼び出す。
   - トークンはメモリのみに保持し、localStorage/sessionStorage には永続化しない。

5. **マルチアカウントログイン**
   - クライアントはメモリ上で複数のセッション（アカウント毎）を保持できる。
   - ログイン毎に独立したセッションペア（アクセストークン＋リフレッシュトークン）を生成する。
   - セッションリストはドメイン状態で管理する。
   - ログアウトは選択されたセッションのみに影響する。

6. **アカウント切り替え**
   - 複数セッションがアクティブな場合、UI にアカウント切り替え機能を提供する。
   - 切り替え時は再認証不要で、API 呼び出しに使用するアクティブな bearer トークンを変更する。
   - アクティブセッションの選択はメモリのみで保持する（永続化しない）。

7. **デバイス/セッション管理**
   - ログイン中のセッション（デバイス）を一覧表示できるようにする。
   - 特定のセッションをリモートで無効化（ revoke ）できるようにする。
   - 「他のすべてのデバイスをログアウト」機能を提供し、一括で無効化できるようにする。
   - セッションメタデータにはデバイス名（User-Agent 由来）、ログイン時刻、最終アクティブ時刻、IP ハッシュを含める。
   - セッション操作は自分のアカウントのセッションに限定し、現在のセッションも「他のデバイス無効化」の対象に含める。

**BREAKING**

- 既存の opaque bearer トークン形式が JWT 形式に変更される。外部連携がトークン内容に依存していた場合、影響を受ける可能性がある。
- `Authorization: Bearer <token>` ヘッダーの形式は変わらないが、トークン自体の解釈が変わる。

## Spec Units

### New Spec Units

（該当なし。本変更は既存 Spec Unit の拡張）

### Modified Spec Units

- `auth-be`: 既存の認証バックエンドを拡張し、JWT 発行・検証、リフレッシュトークンの発行・ローテーション・TTL 管理、マルチセッション対応の Valkey ストレージスキーマ、および保護エンドポイントでの JWT 検証ミドルウェアを追加する。
- `auth-fe`: 既存の認証フロントエンドを拡張し、JWT デコード・有効期限監視、自動リフレッシュ、メモリ上のマルチアカウントセッション管理、アカウント切り替え UI の提供を追加する。

- BE Scenario IDs: `AUTH-BE-S###`

## Impact

- **Backend (`packages/backend`)**: JWT 署名・検証ロジック、リフレッシュトークン管理、新規 `/api/v1/auth/refresh` エンドポイント、Valkey キースキーマ拡張、Config.toml スキーマ変更、セッション一覧・無効化エンドポイント、セッションメタデータ管理。
- **Frontend (`packages/frontend/app`, `packages/frontend/domain`)**: トークン有効期限監視、リフレッシュロジック、マルチセッション状態管理、アカウント切り替え UI、デバイス管理ページ・コンポーネント。
- **API Contract (`packages/typespec`)**: `POST /api/v1/auth/refresh` の追加、JWT bearer スキーマのドキュメント化、`GET /api/v1/sessions` と `DELETE /api/v1/sessions/{id}` と `DELETE /api/v1/sessions/others` の追加。
- **Config / Operations**: `auth.refresh_token_ttl` の運用設定追加。
- **Security**: リフレッシュトークンの盗難リスクに対する検出（ローテーション失敗時の無効化）が必要。リモートセッション無効化能力により、紛失デバイスや不審なアクティビティへの対応が可能になる。セッションメタデータ（IP ハッシュなど）の取り扱いにはプライバシー考慮が必要。
- **Performance**: JWT 検証は各リクエストで実行されるが、署名アルゴリズム（HS256 or RS256）の選択によりオーバーヘッドは軽微。セッション一覧取得は Valkey のセット操作で行い、オーバーヘッドは最小限。
