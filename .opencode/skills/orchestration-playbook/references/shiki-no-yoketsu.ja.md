# 指揮の要訣をエージェント組織へ導入する

この文書は提案の背景資料。
運用の型は `SKILL.md` に集約する。

## 定義

指揮の要訣は二項で表す。

- 確実に掌握する
- 明確な企図のもとで適時適切な命令を与え、行動を律する

補助原則。

- 統制は必要最小限にとどめ、自主裁量の余地を与える
- 掌握のために統御、現況把握、実行の監督を重視する

トップダウンの指揮を前提に置く。
同時に実行側の自律も求める。

## 要点

### 確実な掌握

掌握は統制を薄くする前提。

- 人
  - 能力
  - 信頼度
  - 作業速度
  - 期待品質
- 仕事
  - 任務の目的
  - 期限
  - 重要度
  - 依存関係
- 手段
  - 利用できる道具
  - 予算
  - 権限
  - 制約

掌握が曖昧なまま統制を厚くすると誤差が増える。
掌握が固いほど統制は薄くなる。

### 明確な企図

企図は上位意図と到達点を短い文で固定する。

企図が明確。

- 部下は手段を選べる
- 想定外でも判断が止まらない
- 優先順位が揺れにくい

企図が曖昧。

- 命令が細部へ流れる
- 監督が作業指示へ崩れる
- 自主裁量が暴走か停止へ偏る

### 適時適切な命令

命令は正しさだけでなくタイミングも勝敗を分ける。

- 早すぎる命令は情報不足の拘束を生む
- 遅すぎる命令は好機を逃す
- 長すぎる命令は解釈の分岐を増やす

命令は次を中心に置く。

- 目的
- 境界
- 期待成果物
- 期限

手順は必要最小限に絞る。

### 行動を律する

律は個別作業指示ではなく行動の型。

- 優先順位の型
- 報告の型
- 例外時の型
- 委任範囲の型

律が整うと監督は減る。
律が崩れると命令が増える。

### 統制は必要最小限

統制は安全装置。操縦桿にしない。

- 何をさせないか
- 何を任せるか
- どこで止めるか

統制が薄いと逸脱する。
統制が厚いと機動力が落ちる。

### 自主裁量の余地

自主裁量は企図の枠内での創意。

- 手段選択を任せる
- 代替案の提示を許す
- 局所最適の裁量を許す

機能条件。

- 企図が短い
- 境界が明示
- 報告が速い
- 監督が結果中心

## 掌握を支える三要素

### 統御

統御は動きを同じ方向へ揃える力。

- 価値観
- 規律
- 信頼
- 役割理解
- 用語の共通化

統御が弱い組織は統制を増やしても崩れる。
統御が強い組織は統制が薄くても動く。

### 現況把握

現況把握は今を素早く知る仕組み。

- 進捗
- 位置
- 余力
- 変化
- リスク

現況把握が遅いと命令も監督も遅れる。
現況把握が速いと命令は短くなる。

### 実行の監督

監督は代行ではない。

- 結果を確認する
- 逸脱を止める
- 学びを残す

監督が手順へ踏み込みすぎると自主裁量が死ぬ。
監督が結果だけへ寄ると破綻の兆候を見落とす。

## 指揮の要訣と分権型

分権型は暗黙の連携へ寄せ、明示的な統制を抑える。
指揮の要訣は指揮官が企図を示し、律を整える。
統制を薄く書いても部下任せへは寄せない。

## エージェントオーケストレーションへの翻訳

### 用語対応

| 原語       | エージェント組織での意味 | 実装の単位     |
| ---------- | ------------------------ | -------------- |
| 指揮官     | オーケストレータ         | 司令プロセス   |
| 指揮下部隊 | サブエージェント群       | ワーカープール |
| 任務       | ゴール                   | 要求仕様       |
| 企図       | インテント               | 意図文書       |
| 命令       | タスク付与               | 指令メッセージ |
| 統制       | ガードレール             | 権限制約       |
| 自主裁量   | 自律実行                 | 裁量枠         |
| 統御       | 整合と信頼               | 共有規範       |
| 現況把握   | 状態観測                 | テレメトリ     |
| 実行監督   | 評価と停止               | レビューと中断 |

### 導入の狙い

- 企図で方向を固定する
- 統制を薄く保ち、逸脱を止める
- 現況把握と監督で修正を早める

## 実装パターン

### 企図を先に固定

最初に意図文書を作り、全員へ配る。
意図文書は短く保つ。

最小構成。

- 目的
- 到達点
- 優先順位
- 禁止事項
- 期限
- 受け入れ条件

### 命令はミッション型

命令は手順より成果で書く。
全分岐の列挙は避ける。

### 統制は封筒で表現

統制は動ける封筒を定義する。

- 利用可能な道具
- 外部への書き込み可否
- 金額上限
- 時間上限
- 反復上限
- 人手確認が必要な境界

封筒の設計が良いほど命令と監督が減る。

### 現況把握は標準報告

報告は自由形式にしない。
短い定型へ揃える。

- いまどこまで進んだか
- 次に何をするか
- 何が詰まっているか
- 何の判断が要るか
- リスクは何か

### 監督は二段

- 自動監督
  - 制約違反の検知
  - 予算と時間の超過
  - 依存関係の破綻
- 人の監督
  - 重要な意思決定
  - 破壊的変更
  - 外部公開

監督は停止権限と組で運用する。

## オーケストレータの指揮サイクル

1 観測

- 状態
- 報告
- 外部環境

2 方向付け

- 企図へ照合
- 優先順位を更新
- リスクを評価

3 決心

- 追加命令
- 再配分
- 停止

4 行動

- 指令送付
- 封筒の調整
- 監督設定

循環を短く保つほど適時の命令に寄る。

## 導入手順

段階 0 組織設計

- エージェントの役割定義
- 能力の登録
- 失敗時の責任境界

段階 1 意図文書の標準化

- 企図テンプレート
- 受け入れ条件
- 禁止事項

段階 2 命令フォーマットの統一

- ミッション型の指令
- 期限と成果物
- 封筒の明示

段階 3 現況把握の計測

- 状態ストア
- 進捗イベント
- 重要メトリクス

段階 4 監督の運用

- 自動停止
- 人手レビュー
- 事後分析

## 具体例

### バグ調査と修正

企図。

- 目的
  - 障害の再発を止める
- 到達点
  - 再現手順と原因が一致し、修正をテストで担保
- 禁止事項
  - 本番へ直接変更を当てない

命令。

- 調査班
  - ログと差分から原因候補を三つに絞る
- 再現班
  - 最小再現を作る
- 修正班
  - 修正案を二つ作り副作用を比較する

監督。

- 重大な設計変更は人が承認
- 期限を超えたら中断し方針を更新

### 仕様策定

企図。

- 目的
  - 実装者が迷わず着手できる仕様へ揃える
- 到達点
  - API と UI と状態遷移が整合

命令。

- 仕様班
  - 最小機能の境界を決める
- 検証班
  - 想定外ケースを洗い出す
- 表現班
  - 用語を揃え、読める形へ整える

## チェックリスト

指揮官側。

- 企図が短い
- 命令が成果中心
- 統制が封筒になっている
- 現況把握の線がある
- 止める権限がある

部下側。

- 企図を復唱できる
- 裁量範囲を理解している
- 報告が定型で速い
- 代替案を出せる

## よくある失敗

- 企図が説明書になる
- 命令が手順書になる
- 統制が増え続ける
- 現況把握が遅くなる
- 監督が作業の代行へ崩れる

崩れの兆候として扱い、型へ戻す。
